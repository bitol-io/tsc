## Summary

This RFC introduces OMDS (Open Metadata Difference Standard), a YAML-based format (`kind: MetadataDifference`) for describing the differences between two versions of an ODCS data contract. OMDS standardizes how tools exchange, display, and act upon detected changes, including semantic versioning impact classification aligned to SemVer.

## Motivation

When a data contract evolves, stakeholders need to understand what changed, whether changes are breaking, and what semantic version bump is appropriate. Today, diff output is tool-specific and not interoperable. A standardized difference format enables:

- **Automated CI/CD gates**: reject or flag breaking changes before merge.
- **Change audit trails**: persist diffs as first-class artifacts alongside contracts.
- **Cross-tool interoperability**: any tool that reads ODCS can also produce or consume a standardized diff.
- **Human review**: a structured, readable diff reduces the risk of missed breaking changes.

This aligns with the guiding value of *favoring interoperability over readability* â€” the format is designed for tool exchange first, human consumption second.

## Design and examples

### Top-level structure

An OMDS document is a YAML file with the following top-level fields:

| Field              | Type   | Required | Description                                                               |
| ------------------ | ------ | -------- | ------------------------------------------------------------------------- |
| `apiVersion`       | string | yes      | OMDS version, e.g. `v1.0.0`                                               |
| `kind`             | string | yes      | Always `MetadataDifference`                                               |
| `sourceKind`       | string | yes      | The kind of documents being compared, e.g. `DataContract`                 |
| `source1`          | object | yes      | Reference to the first (baseline) document                                |
| `source2`          | object | yes      | Reference to the second (target) document                                 |
| `scope`            | string | no       | Section filter applied, e.g. `schema`, `slaProperties`. Defaults to `all` |
| `impact`           | object | yes      | Aggregated SemVer impact counts                                           |
| `suggestedVersion` | string | no       | Recommended next semantic version based on `source1.version` and `impact` |
| `diffs`            | array  | yes      | List of individual difference entries                                     |
| `customProperties` | array  | no       | Extension point for tool-specific metadata                                |

### Source reference

Each `source1` and `source2` object identifies one side of the comparison:

| Field          | Type          | Required | Description                      |
| -------------- | ------------- | -------- | -------------------------------- |
| `id`           | string (UUID) | yes      | Document identifier              |
| `version`      | string        | yes      | Semantic version of the document |
| `canonicalUrl` | string        | no       | Permalink to the document        |

### Impact summary

The `impact` object aggregates the SemVer impact across all diffs:

| Field   | Type    | Required | Description                                |
| ------- | ------- | -------- | ------------------------------------------ |
| `major` | integer | yes      | Count of breaking changes                  |
| `minor` | integer | yes      | Count of backward-compatible changes       |
| `patch` | integer | yes      | Count of cosmetic / non-functional changes |

The suggested version is computed by applying the highest impact level to `source1.version` using standard SemVer arithmetic: any major > 0 bumps major; else any minor > 0 bumps minor; else bumps patch.

### Diff entry

Each element in the `diffs` array describes a single atomic difference:

| Field         | Type   | Required | Description                                                                                         |
| ------------- | ------ | -------- | --------------------------------------------------------------------------------------------------- |
| `type`        | string | yes      | One of `ADD`, `REMOVE`, `CHANGE`                                                                    |
| `section`     | string | yes      | Top-level section where the change occurred (e.g. `schema`, `description`, `team`, `slaProperties`) |
| `field`       | string | yes      | Leaf field name that changed                                                                        |
| `path`        | string | yes      | Path to the changed element (see _Path notation_ below)                                             |
| `value1`      | any    | no       | Value from source1 (absent for `ADD`)                                                               |
| `value2`      | any    | no       | Value from source2 (absent for `REMOVE`)                                                            |
| `message`     | string | no       | Human-readable description of the change                                                            |
| `impactLevel` | string | yes      | SemVer impact: `MAJOR`, `MINOR`, `PATCH`, or `UNKNOWN`                                              |

### Path notation

> **Open discussion item.** The path notation should use one of:
>
> - **JSON Pointer (RFC 6901)**: e.g. `/schema/0/properties/2/physicalName` â€” standard, well-supported, but positional array indices are fragile when arrays are unordered.
> - **JSONPath**: e.g. `$.schema[?(@.name=='Address')].properties[?(@.name=='postal_code')].physicalName` â€” more expressive, supports named lookups, but more complex and less widely standardized.
> - **Hybrid**: JSON Pointer extended with named-key selectors for unordered arrays, e.g. `/schema[name=Address]/properties[name=postal_code]/physicalName` â€” readable and stable, but non-standard.
>
> The TSC should decide which notation to adopt. The hybrid approach is what the reference implementation currently uses.

### Impact classification rules (ODCS)

The impact level for each diff depends on the section and the type of change. For `DataContract` documents:

**Schema section:**

| Change type | Fields                                                                                                                      | Impact  |
| ----------- | --------------------------------------------------------------------------------------------------------------------------- | ------- |
| `REMOVE`    | any                                                                                                                         | `MAJOR` |
| `CHANGE`    | `name`, `physicalName`, `logicalType`, `physicalType`, `required`, `unique`, `criticalDataElement`                          | `MAJOR` |
| `CHANGE`    | `quality`                                                                                                                   | `MINOR` |
| `CHANGE`    | other fields                                                                                                                | `PATCH` |
| `ADD`       | `properties`, `name`, `physicalName`, `logicalType`, `physicalType`, `required`, `unique`, `criticalDataElement`, `quality` | `MINOR` |
| `ADD`       | other fields                                                                                                                | `PATCH` |

**All other sections:**

| Change type | Fields              | Impact  |
| ----------- | ------------------- | ------- |
| `REMOVE`    | any                 | `MINOR` |
| `CHANGE`    | `contractCreatedTs` | `PATCH` |
| `CHANGE`    | other fields        | `MINOR` |
| `ADD`       | any                 | `PATCH` |

> **Open discussion item.** These rules are derived from the reference implementation. The TSC should decide whether impact classification rules should be part of the OMDS standard itself (normative) or remain tool-specific guidance (informative).

### Inclusion of raw values (`value1` / `value2`)

> **Open discussion item.** The reference implementation includes the full JSON values from both sides of each diff. This enables tools to render rich comparisons, but increases payload size and may expose sensitive data. The TSC should decide whether `value1`/`value2` are:
>
> - **Required**: always present (absent only when logically inapplicable, i.e. `value1` on `ADD`, `value2` on `REMOVE`).
> - **Optional**: tools may include them, consumers must not depend on them.
> - **Excluded**: only path, type, and message are standardized.

### Example 1: Minimal

A single-field rename detected between two versions of a contract:

```yaml
apiVersion: v1.0.0
kind: MetadataDifference
sourceKind: DataContract
source1:
  id: "34cae6d7-7648-38b2-8f66-8db79e1e2ce4"
  version: "0.1.0"
source2:
  id: "34cae6d7-7648-38b2-8f66-8db79e1e2ce4"
  version: "0.1.6"
scope: all
impact:
  major: 1
  minor: 0
  patch: 0
suggestedVersion: "1.0.0"
diffs:
- type: CHANGE
  section: schema
  field: name
  path: "/schema[name=Address]/properties[name=postal_code]/name"
  value1: "postal_code"
  value2: "zip"
  message: "Renamed property from postal_code to zip"
  impactLevel: MAJOR
```

### Example 2: Multiple changes across sections

A more realistic diff with schema additions, field changes, and metadata updates:

```yaml

sourceKind: DataContract

  minor: 2
  patch: 1

- type: REMOVE
  section: schema
  field: properties
  path: "/schema[name=Customer]/properties[name=phone]"
  value1:
    name: "phone"
    physicalName: "phone"
    logicalType: "string"
    physicalType: "VARCHAR (20)"
  message: "Removed property phone from Customer"
  impactLevel: MAJOR
- type: ADD
  section: schema
  field: properties
  path: "/schema[name=Customer]/properties[name=mobile_phone]"
  value2:
    name: "mobile_phone"
    physicalName: "mobile_phone"
    logicalType: "string"
    physicalType: "VARCHAR (20)"
  message: "Added property mobile_phone to Customer"
  impactLevel: MINOR
- type: CHANGE
  section: schema
  field: quality
  path: "/schema[name=Customer]/properties[name=email]/quality"
```

## ODPS support

OMDS applies to both ODCS and ODPS. This section describes how the standard adapts to `sourceKind: DataProduct` and the design decisions made for each structural difference.

### Port-scoped paths

ODPS structures data through `outputPorts` and `inputPorts`, each identified by the composite key `name` + `version`. Diff paths use the same notation as ODCS but with port-level scoping:

- `/outputPorts[name=rawtransactions,version=2.0.0]/contractId` â€” identifies a specific port version
- `/inputPorts[name=payments,version=1.0.0]/contractId` â€” identifies a specific input port version
- `/managementInterface[name=tpc-dict-update]/description` â€” identifies a management interface entry

The composite key `[name=X,version=Y]` is necessary because ODPS allows multiple entries with the same `name` but different `version` values in both `inputPorts` and `outputPorts`.

### Envelope-only diffing

OMDS for ODPS diffs the **product envelope only** â€” it does not transitively resolve or diff referenced ODCS contracts. When a port's `contractId` changes, the diff records the `contractId` change as a single entry. Consumers who need the schema-level detail should produce a separate OMDS document (`sourceKind: DataContract`) comparing the two referenced contracts.

This keeps OMDS self-contained: an OMDS document requires only the two source documents, not transitive resolution of external references. A tool may optionally include a `relatedDiffs` custom property linking to companion OMDS documents for the underlying contracts:

```yaml
customProperties:
- property: relatedDiffs
  value:
  - sourceKind: DataContract
    source1:
      id: "c2798941-1b7e-4b03-9e0d-955b1a872b32"
      version: "1.0.0"
    source2:
      id: "c2798941-1b7e-4b03-9e0d-955b1a872b33"
      version: "2.0.0"
```

### Port identity

Ports are identified by their `name`. When comparing two ODPS versions:

- **Same name, same version, different content**: standard `CHANGE` diff on the modified fields.
- **Same name, new version added**: `ADD` diff for the new port version entry.
- **Same name, version removed**: `REMOVE` diff for the removed port version entry.
- **Name removed entirely** (all versions gone): one `REMOVE` diff per version entry that existed.
- **New name appears**: one `ADD` diff per version entry.
- **Same name, contractId replaced** (different UUID, not just different version): recorded as a `CHANGE` on `contractId`. The impact is `MAJOR` because the port's promise has fundamentally changed â€” consumers bound to the old contract cannot assume compatibility.

### Port versioning and `suggestedVersion`

OMDS computes `suggestedVersion` at the **product level only** (based on `source1.version`). Port-level version changes are recorded as diffs but do not independently produce a suggested port version â€” that is the responsibility of the port's referenced ODCS contract diff.

> **Open discussion item.** Should OMDS also compute per-port suggested versions? This would require an `impact` object per port, significantly increasing complexity. The current proposal keeps `suggestedVersion` product-scoped.

### Team structure

ODPS `team` is a flat array of members (same structure as ODCS per RFC-0010). Both use the same path notation: `/team[name=Jean-Georges Perrin]/role`. No special handling is needed.

### Impact classification rules (ODPS)

The impact level for each diff depends on the section and the type of change. For `DataProduct` documents:

**Output ports section (`outputPorts`):**

| Change type | Fields                                                              | Impact  | Rationale                                             |
| ----------- | ------------------------------------------------------------------- | ------- | ----------------------------------------------------- |
| `REMOVE`    | port version entry                                                  | `MAJOR` | Consumers lose access to a promised dataset           |
| `CHANGE`    | `contractId` (different UUID)                                       | `MAJOR` | Underlying schema contract replaced entirely          |
| `CHANGE`    | `contractId` (same UUID, version change only via contract metadata) | `MINOR` | Schema evolved but contract identity preserved        |
| `CHANGE`    | `name`                                                              | `MAJOR` | Port identity changed â€” consumers' references break   |
| `CHANGE`    | `type`                                                              | `MAJOR` | Output format changed (e.g. `tables` to `topic`)      |
| `CHANGE`    | `description`                                                       | `PATCH` | Documentation only                                    |
| `CHANGE`    | `sbom`                                                              | `PATCH` | Implementation metadata, not consumer-facing          |
| `CHANGE`    | `inputContracts`                                                    | `MINOR` | Lineage changed but output promise unchanged          |
| `ADD`       | port version entry                                                  | `MINOR` | New version of an existing port â€” backward compatible |
| `ADD`       | new port name                                                       | `MINOR` | New capability added                                  |
| `ADD`       | other fields                                                        | `PATCH` | Metadata enrichment                                   |

**Input ports section (`inputPorts`):**

| Change type | Fields       | Impact  | Rationale                                                             |
| ----------- | ------------ | ------- | --------------------------------------------------------------------- |
| `REMOVE`    | port entry   | `MINOR` | Input dependency removed â€” may affect lineage but not consumer-facing |
| `CHANGE`    | `contractId` | `MINOR` | Source changed but output ports are the consumer contract             |
| `ADD`       | port entry   | `PATCH` | New input source â€” no consumer impact                                 |

**Product metadata:**

| Change type | Fields                                  | Impact  | Rationale                                              |
| ----------- | --------------------------------------- | ------- | ------------------------------------------------------ |
| `CHANGE`    | `status` (to `deprecated` or `retired`) | `MAJOR` | Product lifecycle signal â€” consumers must act          |
| `CHANGE`    | `status` (other transitions)            | `MINOR` | Lifecycle progression                                  |
| `CHANGE`    | `domain`                                | `MINOR` | Organizational change, may affect discoverability      |
| `CHANGE`    | `name`                                  | `MINOR` | Display name change, `id` remains the stable reference |
| `CHANGE`    | `productCreatedTs`                      | `PATCH` | Timestamp metadata                                     |
| `CHANGE`    | `description`                           | `PATCH` | Documentation only                                     |
| `CHANGE`    | `tags`                                  | `PATCH` | Discovery metadata                                     |
| `REMOVE`    | any metadata field                      | `MINOR` | Information loss                                       |
| `ADD`       | any metadata field                      | `PATCH` | Metadata enrichment                                    |

**Management interface (`managementInterface`):**

| Change type | Fields                | Impact  | Rationale                     |
| ----------- | --------------------- | ------- | ----------------------------- |
| `REMOVE`    | interface entry       | `MINOR` | Management capability removed |
| `CHANGE`    | `url`, `channel`      | `MINOR` | Access point changed          |
| `CHANGE`    | `description`, `tags` | `PATCH` | Documentation only            |
| `ADD`       | interface entry       | `PATCH` | New management capability     |

**Support and team sections:** same rules as ODCS "all other sections" (REMOVE = MINOR, CHANGE = MINOR, ADD = PATCH).

## Alternatives

### 1. Use JSON Patch (RFC 6902) directly

JSON Patch is a well-established standard for describing transformations between two JSON documents. However, it is operation-oriented (designed to *apply* changes), not analysis-oriented. It lacks semantic versioning impact, section awareness, and human-readable messages. OMDS could reference JSON Patch paths but needs its own envelope for impact analysis.

Decision: rejected.

### 2. Embed diffs in the contract itself

Rather than a separate document kind, diffs could be stored as a `changelog` section within the contract. This was rejected because: (a) it couples diff history to the contract lifecycle, (b) it inflates contract size over time, and (c) diffs between contracts owned by different teams should not require write access to either contract.

Decision: rejected, too complicated to analyze.

### 3. Free-form text diffs (unified diff format)

Line-oriented diffs (like `diff -u`) are universally understood but lose structural awareness â€” they cannot distinguish a breaking schema change from a cosmetic description update. OMDS preserves YAML structure awareness.

Decision: might be too free-form?

## Open issues and discussion

This section consolidates all items requiring TSC discussion before approval.

### OI-1: Path notation

The path notation should use one of three approaches:

- **Option A â€” JSON Pointer (RFC 6901)**: e.g. `/schema/0/properties/2/physicalName`. Standard and well-supported, but positional array indices are fragile when arrays are unordered (which is the case for ODCS `schema` and ODPS `outputPorts`).
- **Option B â€” JSONPath (RFC 9535)**: e.g. `$.schema[?(@.name=='Address')].properties[?(@.name=='postal_code')].physicalName`. More expressive with named lookups, but more complex and less widely adopted.
- **Option C â€” Hybrid**: JSON Pointer extended with named-key selectors for unordered arrays, e.g. `/schema[name=Address]/properties[name=postal_code]/physicalName`. Readable and stable, but non-standard. This is what the reference implementation currently uses.

**Recommendation**: Option C (hybrid) â€” it is human-readable, stable across reordering, and already proven in the reference implementation.

### OI-2: Impact classification rules â€” normative or informative?

The ODCS and ODPS impact classification tables in this RFC are derived from the reference implementation. The TSC should decide whether these rules are:

- **Normative**: all OMDS-producing tools MUST follow these tables (tools may be stricter but not more lenient).
- **Informative**: the tables are guidance; tools may assign impact levels differently.

**Recommendation**: Normative â€” inconsistent impact classification across tools would undermine interoperability (a `PATCH` in one tool and `MAJOR` in another for the same change defeats the purpose of a standard).

### OI-3: Inclusion of raw values (`value1` / `value2`)

The reference implementation includes the full JSON values from both sides of each diff. This enables tools to render rich comparisons, but increases payload size and may expose sensitive data. Options:

- **Required**: always present (absent only when logically inapplicable, i.e. `value1` on `ADD`, `value2` on `REMOVE`).
- **Optional**: tools may include them, consumers must not depend on them.
- **Excluded**: only path, type, and message are standardized.

**Recommendation**: Optional â€” this gives tool authors flexibility without burdening lightweight implementations. The `path` + `type` + `message` fields provide enough information for most consumers.

### OI-4: Per-port suggested versions for ODPS

Currently `suggestedVersion` is computed at the product level only. Should OMDS also compute per-port suggested versions? This would require an `impact` object per port, adding significant structural complexity.

**Recommendation**: Product-level only for v1.0.0. Per-port versions can be added in a future OMDS revision if demand warrants it.

### OI-5: Scope of `sourceKind`

Currently OMDS defines behavior for `DataContract` and `DataProduct`. Should the standard be open-ended (any `kind` value, with impact rules left to tool implementers for unknown kinds) or closed (only explicitly defined kinds are valid)?

**Recommendation**: Closed for normative impact rules, open for structural diffing. A tool may produce an OMDS document for any YAML `kind`, but `impactLevel` MUST be `UNKNOWN` for kinds without a defined classification table in this standard.

### OI-6: Alternative 3 â€” free-form text diffs

The user noted "might be too free-form?" for Alternative 3 (unified diff format). The TSC should confirm rejection. Line-oriented diffs lose structural awareness â€” they cannot distinguish a breaking schema change from a cosmetic description update, and they are not machine-actionable for CI/CD gating.

## Decision

> The decision made by the TSC.

## Consequences

- Tools producing ODCS or ODPS diffs should adopt the `MetadataDifference` kind for interoperability.
- The reference implementation in `ai.jgp.bitol.svc` will need to emit OMDS-compliant YAML in addition to its current JSON response, and extend its comparator to support `DataProduct` documents.
- Impact classification rules, once standardized, constrain how tools assign severity â€” tools may be stricter (classify more changes as MAJOR) but not more lenient than the standard.
- ODPS diffs are envelope-only by design. Tools that need schema-level detail must produce companion OMDS documents for the referenced ODCS contracts.
