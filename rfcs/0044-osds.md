## Summary

This RFC introduces OSDS (Open Semantic Standard), a YAML-based format (`kind: SemanticDefinition`) for describing reusable semantic definitions (concepts, their semantic types, their relationships, and their authoritative sources) that ODCS contracts and ODPS products reference instead of re-defining meaning inline.

OSDS is deliberately **federated**: one file is a self-contained semantic layer, and a constellation of cross-referencing files, each owned by a domain, forms the enterprise semantic layer. There is no central model. Governance lives in the interoperability rules (how concepts are identified and referenced), aligned with the federated computational governance of Data Mesh.

OSDS reuses existing Bitol building blocks wherever possible: the shared Authoritative Definitions block (for outward links to ontologies and glossaries), the ODCS reference notation (for stable, refactor-safe links), and the ODCS relationship model (concept-level and document-level placement). The only genuinely new surface is the `definitions` block, a `semanticType` field, an open relationship-`type` vocabulary, and one new recommended Authoritative Definitions value (`semanticDefinition`) for binding from contracts and products.

## Motivation

The industry is converging on the idea of a "semantic layer," but most implementations (LookML, Cube, dbt MetricFlow, AtScale) are centralized and tool-specific. A central semantic tier becomes a coupling point, a bottleneck, and a governance nightmare: the opposite of the autonomy data-product teams need.

Meanwhile, Bitol already has the right primitives to express meaning, but they are trapped *inside* individual contracts and products: every ODCS contract re-types the same `logicalType`, re-writes the same description, and re-links the same glossary entry for what is, semantically, the same concept. There is no addressable, reusable definition of "Customer" or "Customer Lifetime Value" that multiple contracts and products can share.

OSDS standardizes that shared, federated definition layer. It enables:

- **Reuse across artifacts**: a concept is defined once, in its owning domain, and referenced by any number of ODCS contracts and ODPS products.
- **Federated governance**: each domain owns and versions its own semantic file; cross-domain agreement is expressed by *links* (`sameAs`, `broaderThan`), not by a shared master schema.
- **Interoperability with the semantic web**: OSDS links *out* to RDF/OWL/SKOS ontologies through Authoritative Definitions rather than reinventing them, so existing ontologies remain authoritative.
- **No churn in existing standards**: binding from ODCS/ODPS rides on the already-shared Authoritative Definitions block, requiring no schema change to either standard.

This aligns with the guiding value of *favoring interoperability over readability*: identity is stable and refactor-safe first, human-friendly second.

## Design and examples

Each construct below is introduced with a simple example, then a more complex example, and finally a field table describing what it does.

### Top-level structure (envelope)

An OSDS document is a YAML file. A minimal document carries just an envelope and one definition:

```yaml
apiVersion: v1.0.0
kind: SemanticDefinition
id: 7c9e6679-7425-40de-944b-e07fc1f90ae7
name: Sales Semantics
domain: sales
definitions:
  - id: customer-lifetime-value
    name: Customer Lifetime Value
```

A more complete envelope adds lifecycle, versioning, document-level authoritative sources, and shared blocks:

```yaml
apiVersion: v1.0.0
kind: SemanticDefinition
id: 7c9e6679-7425-40de-944b-e07fc1f90ae7
name: Sales Semantics
domain: sales
status: active
version: 1.2.0
description:
  purpose: Shared semantic definitions owned by the Sales domain.
authoritativeDefinitions:
  - type: glossary
    url: https://wiki.acme/sales/glossary
tags:
  - sales
  - semantics
definitions:
  - id: customer
    name: Customer
  - id: customer-lifetime-value
    name: Customer Lifetime Value
customProperties:
  - property: steward
    value: jane.doe@acme.com
```

| Field                      | Type   | Required | Description                                                                              |
| -------------------------- | ------ | -------- | ---------------------------------------------------------------------------------------- |
| `apiVersion`               | string | yes      | Version of the standard used. Default `v1.0.0`.                                          |
| `kind`                     | string | yes      | Always `SemanticDefinition`.                                                             |
| `id`                       | string | yes      | Unique identifier for the document, such as a UUID.                                      |
| `name`                     | string | no       | Human-readable name of the semantic document.                                            |
| `domain`                   | string | no       | Owning domain, the unit of federation, e.g. `sales`.                                     |
| `status`                   | string | no       | Lifecycle: `proposed`, `draft`, `active`, `deprecated`, `retired`.                       |
| `version`                  | string | no       | Version of this semantic document.                                                       |
| `description`              | object | no       | High-level description (reuses the ODCS description block).                              |
| `definitions`              | array  | yes      | List of semantic concept definitions. See below.                                         |
| `authoritativeDefinitions` | array  | no       | Document-level external sources of truth (shared Bitol Authoritative Definitions block). |
| `tags`                     | array  | no       | Tags (shared Bitol block).                                                               |
| `customProperties`         | array  | no       | Additional metadata (shared Bitol block).                                                |

### The `definitions` block (the concept)

Each entry in `definitions` defines a single semantic concept. A simple concept names a meaning and its semantic type:

```yaml
definitions:
  - id: customer-lifetime-value
    name: Customer Lifetime Value
    description: Net margin expected over the customer relationship.
    semanticType: monetaryAmount
```

A more complex concept adds an alias, outward links to ontology and glossary, an internal relationship, and shared blocks:

```yaml
definitions:
  - id: customer-lifetime-value
    name: Customer Lifetime Value
    businessName: CLV
    description: Net margin expected over the customer relationship.
    semanticType: monetaryAmount
    authoritativeDefinitions:
      - type: ontology
        url: https://schema.org/MonetaryAmount
      - type: businessDefinition
        url: https://wiki.acme/clv
    relationships:
      - to: definitions/customer
        type: derivedFrom
    tags:
      - kpi
    customProperties:
      - property: unit
        value: USD
```

| Field                      | Type   | Required | Description                                                                                                                       |
| -------------------------- | ------ | -------- | --------------------------------------------------------------------------------------------------------------------------------- |
| `id`                       | string | yes      | Stable, refactor-safe identifier, unique within `definitions`. Pattern `^[A-Za-z0-9_-]+$`. Used as the reference target.          |
| `name`                     | string | yes      | Human-readable concept name, e.g. "Customer Lifetime Value".                                                                      |
| `description`              | string | no       | The business meaning of the concept.                                                                                              |
| `semanticType`             | string | no       | The semantic class of the concept, e.g. `monetaryAmount`, `identifier`, `quantity`, `temporal`, `category`, `party`. Open string. |
| `businessName`             | string | no       | Optional display/alias name distinct from `name`.                                                                                 |
| `relationships`            | array  | no       | Links to other concepts, same document or external. See below.                                                                    |
| `authoritativeDefinitions` | array  | no       | Outward links to ontology/glossary/taxonomy (shared block; `ontology`, `glossary`, `taxonomy`, `businessDefinition`).             |
| `tags`                     | array  | no       | Tags (shared block).                                                                                                              |
| `customProperties`         | array  | no       | Extra metadata (shared block).                                                                                                    |

`semanticType` is intentionally left open in v1 (no fixed enum), suggestions might appear in a later RFC once usage patterns emerge.

### Relationships

A simple relationship is declared at the concept level, where `from` is implicit (the enclosing concept):

```yaml
definitions:
  - id: customer-lifetime-value
    name: Customer Lifetime Value
    relationships:
      - to: definitions/customer
        type: derivedFrom
```

A more complex use combines a cross-domain concept-level link (the federation primitive) with an explicit document-level relationship:

```yaml
definitions:
  - id: customer
    name: Customer
    relationships:
      # cross-domain link: this is the federation primitive
      - to: https://crm.acme/crm.osds.yaml#/definitions/party
        type: sameAs
        description: Equivalent to the CRM domain's Party concept.

# document-level relationship (explicit from + to)
relationships:
  - from: definitions/customer-lifetime-value
    to: definitions/customer
    type: derivedFrom
    customProperties:
      - property: cardinality
        value: one-to-one
```

Relationships reuse the ODCS relationship model (introduced in ODCS v3.1, see RFC-0031), with two placements:

- **Concept level** — declared inside a concept's `relationships`; `from` is implicit (the enclosing concept) and must NOT be specified.
- **Document level** — declared at the root `relationships`; both `from` and `to` are required.

| Field              | Type   | Required          | Description                                                                              |
| ------------------ | ------ | ----------------- | ---------------------------------------------------------------------------------------- |
| `from`             | string | context-dependent | Source concept reference. Required at document level, forbidden at concept level.        |
| `to`               | string | yes               | Target concept reference (slash path, name-dot shorthand, or `<url>#/...` external).     |
| `type`             | string | no                | Relationship type. Open string with recommended values (below). Defaults to `relatedTo`. |
| `description`      | string | no                | Optional description of the relationship.                                                |
| `customProperties` | array  | no                | Extra metadata, e.g. `cardinality: one-to-many`, `label: manages`.                       |

Unlike the ODCS foreign-key relationship (whose `type` is enum-locked to `foreignKey`), the OSDS relationship `type` is an **open string with a recommended-values table**, mirroring how the shared Authoritative Definitions `type` works. This keeps the door open to domain-specific relationship semantics while preserving interoperability for the common cases.

#### Recommended values for relationship `type`

| Value                          | Description                                                   | Maps to (link out via AD)        |
| ------------------------------ | ------------------------------------------------------------- | -------------------------------- |
| `isA`                          | The concept is a specialization of the target.                | `rdfs:subClassOf`                |
| `partOf`                       | The concept is a component/part of the target.                | `dcterms:isPartOf`               |
| `sameAs`                       | The concept is equivalent to the target (often cross-domain). | `owl:sameAs` / `skos:exactMatch` |
| `closeMatch`                   | The concept is approximately equivalent to the target.        | `skos:closeMatch`                |
| `broaderThan` / `narrowerThan` | Hierarchical generalization / specialization.                 | `skos:broader` / `skos:narrower` |
| `derivedFrom`                  | The concept is computed or derived from the target.           | —                                |
| `relatedTo`                    | Generic association (default).                                | `skos:related`                   |

Custom values are allowed; tooling that does not recognize a value should treat the relationship as an opaque labeled link (same rule as Authoritative Definitions `type`).

### Reference notation

References take one of three forms, from simplest to most explicit:

```yaml
# Shorthand (by name)
to: sales.customer_lifetime_value

# Fully qualified (stable, by id)
to: definitions/customer-lifetime-value

# External / federated (self-locating across domains)
to: https://crm.acme/crm.osds.yaml#/definitions/party
```

OSDS reuses the ODCS reference notation verbatim (see the ODCS References documentation). Identity is two-tier, exactly as in ODCS: the document carries a UUID-style `id`, while each concept carries a refactor-safe `StableId` used as the reference target. Because an external reference *locates its own document* (`<url>#/...`), federation needs no central registry — the constellation is self-describing.

### Binding from ODCS and ODPS (no schema change)

A contract property or a product element declares that it *means* an OSDS concept by carrying an Authoritative Definition of a new recommended type, `semanticDefinition`:

```yaml
# inside an ODCS property OR an ODPS element
authoritativeDefinitions:
  - type: semanticDefinition
    url: https://sales.acme/sales.osds.yaml#/definitions/customer-lifetime-value
```

In context, a complete ODCS property is unchanged except for that binding:

```yaml
schema:
  - id: customers_tbl
    name: customers
    properties:
      - id: clv_field
        name: clv
        logicalType: number
        authoritativeDefinitions:
          - type: semanticDefinition
            url: https://sales.acme/sales.osds.yaml#/definitions/customer-lifetime-value
```

Because the Authoritative Definitions block is already shared across all Bitol standards, this binding works in ODCS and ODPS today. The only addition is the recommended `semanticDefinition` value in the shared `type` vocabulary.

## Alternatives

- **A centralized semantic layer / single master model.** Rejected: it reintroduces the coupling and bottleneck that Data Mesh and Bitol exist to avoid. OSDS's constellation-of-files model delivers the same conceptual layer without central ownership.
- **A first-class `semanticRef` field on ODCS/ODPS properties.** Rejected for v1: cleaner UX but it requires schema changes to two standards and creates a second referencing mechanism. Riding on the shared Authoritative Definitions block is more consistent and ships without core changes.
- **Adopting SKOS/OWL verbs as the native relationship vocabulary.** Rejected: it forces authors to learn the semantic web to write a YAML file. Instead OSDS uses plain-English relationship types and *links out* to SKOS/RDF via Authoritative Definitions, preserving interoperability without the learning tax.
- **Referencing concepts by UUID.** Rejected: ODCS deliberately uses stable string ids plus slash paths for references (UUID only at the document root). Following that convention keeps references readable and refactor-safe and keeps OSDS consistent with ODCS.
- **Folding semantics into ODCS itself.** Rejected: it would re-centralize meaning inside individual contracts and prevent reuse by ODPS products and by tools that have neither a contract nor a product.

## Consequences

- A new standard (OSDS) joins the Bitol family alongside ODCS and ODPS, with a small, consistent surface area.
- The shared Authoritative Definitions `type` vocabulary gains one recommended value, `semanticDefinition`. ODCS and ODPS require no schema changes to consume OSDS.
- Tooling can resolve meaning across domains by following self-locating references; a central registry is optional, not required.
- A later RFC may tighten the `semanticType` recommended values and may add a metrics/measures module (relating OSDS to RFC-0034). These are intentionally out of scope for v1.
- Authoring and validation benefit from tooling that resolves stable ids and external references; this is the same tradeoff already accepted by ODCS references.
