# RFC-0037: Open Maturity Model Standard (OMMS)

Champion: Jean-Georges Perrin

Authors: Jean-Georges Perrin

Slack: [#bitol-wg](https://data-mesh-learning.slack.com/archives/C089S376YGM)

GitHub issue: [TSC #51](https://github.com/bitol-io/tsc/issues/51)

## Summary

This RFC proposes the Open Maturity Model Standard (OMMS) v0.2.0, a YAML-based standard for defining maturity models that evaluate Bitol artifacts (data contracts, data products, and potentially others). OMMS aligns with the ODCS data quality vocabulary (`type`, `metric`, `arguments`, operators, `unit`) and introduces `policies` as a first-class concept for expressing organizational rules that override or constrain maturity evaluation.

## Motivation

Bitol now has two artifacts that benefit from maturity assessment: data contracts (ODCS) and data products (ODPS). Both need a structured, measurable way to evaluate how complete, governed, and operational they are. Experimental maturity models have been built for ODCS, confirming the need and validating the approach.

Without a shared standard:

- Maturity model authors have no formal syntax to follow.
- Tooling that evaluates one model cannot generically evaluate another.
- Organizational rules (such as status-based caps on maturity levels) have no formal home.

OMMS v0.2.0 defines a common syntax for maturity models, reuses the ODCS data quality vocabulary for criteria, and creates a foundation for any future maturity model within the Bitol ecosystem or beyond.

This aligns with Bitol's guiding values: open standards, interoperability, and measurability.

## Design and examples

### 1. Document Structure

An OMMS document is a YAML file with the following top-level structure:

```yaml
kind: MaturityModel
apiVersion: v0.2.0

# --- Identity ---
id: <uuid>
name: <string>
version: <semver>
status: <proposed | draft | active | deprecated | retired>

# --- Metadata ---
description:
  purpose: <string>
  limitations: <string>
  usage: <string>
tags: [<string>]
targetApplication: <string>       # e.g., ODCS, ODPS
targetApiVersion: <string>        # e.g., v3.1.0, v1.0.0
customProperties: [...]
authoritativeDefinitions: [...]

# --- Policies ---
policies: [...]

# --- Levels ---
levels: [...]
```

The standard defines the syntax and semantics of each section. It does NOT prescribe level names, level count, or specific criteria. Each maturity model makes those decisions independently.

#### Definitions

| Key                        | Required | Type   | Description                                                             |
| -------------------------- | -------- | ------ | ----------------------------------------------------------------------- |
| `kind`                     | Yes      | string | Must equal `MaturityModel`.                                             |
| `apiVersion`               | Yes      | string | Version of the OMMS standard. Default is `v0.2.0`.                      |
| `id`                       | Yes      | string | A unique identifier (UUID recommended).                                 |
| `name`                     | No       | string | Name of the maturity model.                                             |
| `version`                  | No       | string | Current version of the model (SemVer).                                  |
| `status`                   | Yes      | string | Example values: `proposed`, `draft`, `active`, `deprecated`, `retired`. |
| `description`              | No       | object | Description object containing `purpose`, `limitations`, `usage`.        |
| `description.purpose`      | No       | string | Intended purpose of the maturity model.                                 |
| `description.limitations`  | No       | string | Known limitations.                                                      |
| `description.usage`        | No       | string | How to use the model.                                                   |
| `targetApplication`        | No       | string | The standard this model evaluates (e.g., `ODCS`, `ODPS`).               |
| `targetApiVersion`         | No       | string | Version of the target standard (e.g., `v3.1.0`).                        |
| `policies`                 | No       | array  | List of policies evaluated before level criteria.                       |
| `levels`                   | Yes      | array  | List of maturity levels.                                                |
| `tags`                     | No       | array  | List of tags.                                                           |
| `authoritativeDefinitions` | No       | array  | Shared authoritative definitions structure.                             |
| `customProperties`         | No       | array  | Shared custom properties structure.                                     |

### 2. Policies

Policies are rules evaluated before any level criteria. They express organizational decisions that override or constrain criteria evaluation. A policy can cap the maximum achievable level, enforce conventions, or define validation rules that apply globally.

Policy IDs follow the convention `POL-{X}{nn}` where `{X}` identifies the target artifact:

- `POL-C01`: contract policy
- `POL-P01`: product policy

Each maturity model defines its own policies independently. Policies from different models do not interact.

#### Structure

```yaml
policies:
  - id: <string>           # Required. Format: POL-{X}{nn}
    name: <string>         # Optional. Machine-friendly name
    property: <string>     # Required. The YAML property path
    type: <library | text> # Required
    metric: <string>       # Metric name
    arguments: { ... }     # Optional
    description: <string>  # Optional
```

#### Definitions

| Key           | Required | Type   | Description                                                                 |
| ------------- | -------- | ------ | --------------------------------------------------------------------------- |
| `id`          | Yes      | string | Unique identifier. Suggested format: `POL-{X}{nn}`.                         |
| `name`        | No       | string | Machine-friendly name.                                                      |
| `property`    | Yes      | string | YAML property path to evaluate.                                             |
| `type`        | No       | string | `library` or `text`. Default is `library`                                   |
| `metric`      | No       | string | Metric name (e.g., `validValues`, `rowCount`). Similar to ODCS' DQ metrics. |
| `arguments`   | No       | object | Parameters for the metric (e.g., `caps`, `where`).                          |
| `description` | No       | string | Human-readable explanation.                                                 |
| `tags`                     | No       | array  | List of tags.                                                           |
| `authoritativeDefinitions` | No       | array  | Shared authoritative definitions structure.                             |
| `customProperties`         | No       | array  | Shared custom properties structure.                                     |

#### Example: Status cap policy (ODPS)

```yaml
policies:
  - id: POL-P01
    name: statusCap
    property: status
    type: library
    metric: validValues
    arguments:
      caps:
        retired: 0
        deprecated: 2
        proposed: 1
        draft: 1
        active: 5
    description: >
      Status determines the maximum achievable maturity level.
      Evaluated before all level criteria. The cap is instant
      and absolute.
```

#### Example: Self-sourcing convention policy (ODPS)

```yaml
policies:
  - id: POL-P02
    name: selfSourcing
    property: inputPorts
    type: library
    metric: validValues
    arguments:
      where:
        field: name
        in: [self]
      contractIdMustMatch: product.id
      versionMustMatch: product.version
    description: >
      Data products that generate data without external sources
      must use the "self" convention. When inputPort.name is
      "self", contractId must equal the product id and version
      must equal the product version.
```

### 3. Levels

Levels define the maturity progression. They are cumulative: an artifact must satisfy all criteria of a level and all lower levels to qualify.

OMMS does not prescribe level names, level count, or ordering conventions. Each maturity model makes those decisions independently.

#### Structure

```yaml
levels:
  - level: <integer>          # Required. Level number (1, 2, 3...)
    name: <string>            # Required. Human-readable name
    description: <string>     # Required
    criteria: [...]           # Required. List of criteria
```

#### Definitions

| Key           | Required | Type    | Description                                                                                   |
| ------------- | -------- | ------- | --------------------------------------------------------------------------------------------- |
| `level`       | Yes      | integer | Level number (1, 2, 3...).                                                                    |
| `name`        | Yes      | string  | Human-readable name.                                                                          |
| `description` | Yes      | string  | What this level means. This could be displayed to the end-user. Be cognizant of the verbiage. |
| `criteria`    | Yes      | array   | List of criteria for this level.                                                              |
| `tags`                     | No       | array  | List of tags.                                                           |
| `authoritativeDefinitions` | No       | array  | Shared authoritative definitions structure.                             |
| `customProperties`         | No       | array  | Shared custom properties structure.                                     |

#### Example

```yaml
levels:
  - level: 1
    name: Basic
    description: Minimum valid artifact.
    criteria:
      - id: L1-C01
        property: kind
        type: library
        metric: validValues
        mustBe: DataContract
        description: kind must equal DataContract.

  - level: 2
    name: Enriched
    description: Artifact has business context.
    criteria:
      - id: L2-C01
        property: description.purpose
        type: library
        metric: nullValues
        mustBe: 0
        unit: rows
        description: Purpose is documented.
```

### 4. Criteria

Criteria define the measurable checks within each level. Their syntax aligns with the ODCS data quality vocabulary.

Criteria IDs follow the convention `L{n}-{X}{nn}` where:

- `{n}` is the level number
- `{X}` identifies the target artifact (`C` = contract, `P` = product)
- `{nn}` is a zero-padded sequence within that level

#### Structure

```yaml
- id: <string>                # Required. Format: L{n}-{X}{nn}
  property: <string>          # Required. YAML property path
  type: <library | text>      # Required. library = machine-evaluable,
                              #           text = auto-pass
  metric: <string>            # Metric name
  arguments: { ... }          # Optional. Parameters for the metric
  <operator>: <value>         # Threshold operator
  unit: <rows | percent>      # Optional. Only two valid values
  description: <string>       # Optional
```

#### Definitions

| Key           | Required | Type   | Description                                                                                       |
| ------------- | -------- | ------ | ------------------------------------------------------------------------------------------------- |
| `id`          | Yes      | string | Unique identifier. Format: `L{n}-{X}{nn}`.                                                        |
| `property`    | Yes      | string | YAML property path to evaluate.                                                                   |
| `type`        | No       | string | `library` (machine-evaluable) or `text` (auto-pass). Default is `library`.              |
| `metric`      | No       | string | Metric name. Required when `type` is `library`.                                                   |
| `arguments`   | No       | object | Parameters for the metric.                                                                        |
| `<operator>`  | No       | value  | Threshold operator (`mustBe`, `mustBeGreaterOrEqualTo`, etc.). Required when `type` is `library`. |
| `unit`        | No       | string | `rows` or `percent`.                                                                              |
| `description` | No       | string | Human-readable explanation.                                                                       |
| `tags`                     | No       | array  | List of tags.                                                           |
| `authoritativeDefinitions` | No       | array  | Shared authoritative definitions structure.                             |
| `customProperties`         | No       | array  | Shared custom properties structure.                                     |

##### Arguments

| Key           | Type            | Description                                                                                                               |
| ------------- | --------------- | ------------------------------------------------------------------------------------------------------------------------- |
| `having`      | string or array | Item must have these fields present.                                                                                      |
| `anyOf`       | array           | Item must have at least one of these fields.                                                                              |
| `where`       | object          | Filter items. Contains `field` (string), `in` (array), and/or `isNotNull` (boolean).                                      |
| `distinct`    | boolean         | Count unique values only.                                                                                                 |
| `relatesTo`   | string          | Dependency reference. Bracket notation (`[DataContract]`, `[DataProduct]`) for cross-artifact, plain path for intra-file. |
| `validValues` | array           | Restrict to values in this list.                                                                                          |
| `caps`        | object          | Map status values to maximum achievable levels (policies only).                                                           |

##### Metrics

OMMS reuses the ODCS library metrics. No new metrics are introduced.

| Metric          | Description                          |
| --------------- | ------------------------------------ |
| `rowCount`      | Counts items in a collection         |
| `nullValues`    | Counts null or empty values          |
| `invalidValues` | Counts values not in an allowed set  |
| `validValues`   | Checks values against an allowed set |

##### Threshold Operators

Aligned with ODCS data quality conventions:

| Operator                 | Meaning                    |
| ------------------------ | -------------------------- |
| `mustBe`                 | Exact match                |
| `mustNotBe`              | Must not equal             |
| `mustBeGreaterThan`      | Strict lower bound         |
| `mustBeGreaterOrEqualTo` | Inclusive lower bound      |
| `mustBeLessThan`         | Strict upper bound         |
| `mustBeLessOrEqualTo`    | Inclusive upper bound      |
| `mustBeBetween`          | Inclusive range [min, max] |
| `mustNotBeBetween`       | Exclusive range [min, max] |

The special value `latest` for `mustBe` instructs the evaluator to resolve to the current version of the `targetApplication` standard.

##### Units

OMMS supports two units:

| Unit      | Description        |
| --------- | ------------------ |
| `rows`    | Count of items     |
| `percent` | Percentage (0-100) |

#### Example: Simple field check

**Criterion:**
```yaml
- id: L1-P01
  property: apiVersion
  type: library
  metric: validValues
  mustBe: v1.0.0
  description: ODPS standard version is v1.0.0.
```

**Data product:**
```yaml
apiVersion: v1.0.0
kind: DataProduct
id: 064c4630-8aad-4dc0-ba95-0f69940e6b18
```

**Evaluation:**

1. Read `property`: `apiVersion`
2. Navigate to `apiVersion` in the data product → `v1.0.0`
3. `metric: validValues` + `mustBe: v1.0.0` → does `v1.0.0` equal `v1.0.0`?
4. **Result: PASS**

### 5. Evaluation Walkthrough

The following examples show how an evaluator processes criteria against actual artifacts, step by step.

#### Example 1: Null check with unit: rows

**Criterion:**
```yaml
- id: L2-P02
  property: name
  type: library
  metric: nullValues
  mustBe: 0
  unit: rows
  description: Product name must be present.
```

**Data product (passing):**
```yaml
name: Customer Data Product
```

**Evaluation:**

1. Navigate to `name` → `Customer Data Product`
2. `metric: nullValues` → count null/empty values → 0
3. `mustBe: 0`, `unit: rows` → is 0 == 0?
4. **Result: PASS**

**Data product (failing):**
```yaml
# name is missing
```

**Evaluation:**

1. Navigate to `name` → missing
2. `metric: nullValues` → count null/empty values → 1
3. `mustBe: 0` → is 1 == 0?
4. **Result: FAIL** (actual: 1 row null)

#### Example 2: Collection count

**Criterion:**
```yaml
- id: L1-P05
  property: outputPorts
  type: library
  metric: rowCount
  mustBeGreaterOrEqualTo: 1
  unit: rows
  description: At least one output port is defined.
```

**Data product:**
```yaml
outputPorts:
  - name: rawtransactions
    contractId: c2798941-1b7e-4b03-9e0d-955b1a872b32
  - name: consolidatedtransactions
    contractId: a44978be-1fe0-4226-b840-1b715bc25c63
```

**Evaluation:**

1. Navigate to `outputPorts` → array with 2 items
2. `metric: rowCount` → count = 2
3. `mustBeGreaterOrEqualTo: 1` → is 2 >= 1?
4. **Result: PASS** (actual: 2 rows)

#### Example 3: Nested property with percent

**Criterion:**
```yaml
- id: L2-C04
  property: schema[].properties[].description
  type: library
  metric: nullValues
  mustBeLessThan: 20
  unit: percent
  description: ">= 80% of properties have descriptions (< 20% null)."
```

**Data contract:**
```yaml
schema:
  - name: product
    properties:
      - name: product_id
        description: Unique identifier for the product.
      - name: unit_price
        description: Price per unit in USD.
      - name: sku
      - name: category
        description: Product category.
      - name: created_at
        description: Timestamp when product was created.
```

**Evaluation:**

1. Iterate all properties across all schema objects → 5 properties
2. `metric: nullValues` → count properties where `description` is null/empty → 1 (`sku`)
3. `unit: percent` → (1 / 5) × 100 = 20%
4. `mustBeLessThan: 20` → is 20 < 20?
5. **Result: FAIL** (actual: 20% null, required: < 20%)

#### Example 4: arguments with `having`

**Criterion:**
```yaml
- id: L4-P04
  property: outputPorts[]
  type: library
  metric: rowCount
  arguments:
    having: sbom
  mustBeGreaterOrEqualTo: 1
  unit: rows
  description: At least one output port version has an SBOM declared.
```

**Data product:**
```yaml
outputPorts:
  - name: rawtransactions
    version: 1.0.0
    contractId: c2798941-1b7e-4b03-9e0d-955b1a872b32
  - name: rawtransactions
    version: 2.0.0
    contractId: c2798941-1b7e-4b03-9e0d-955b1a872b33
    sbom:
      - type: external
        url: https://mysbomserver/mysbom
  - name: consolidatedtransactions
    version: 1.0.0
    contractId: a44978be-1fe0-4226-b840-1b715bc25c63
```

**Evaluation:**

1. Iterate all output ports → 3 items
2. `arguments: { having: sbom }` → filter to items that have an `sbom` field → 1 item
3. `metric: rowCount` → count = 1
4. `mustBeGreaterOrEqualTo: 1` → is 1 >= 1?
5. **Result: PASS** (actual: 1 row)

#### Example 5: arguments with `where`

**Criterion:**
```yaml
- id: L3-C10
  property: authoritativeDefinitions
  type: library
  metric: rowCount
  arguments:
    where:
      field: type
      in: [canonicalUrl]
  mustBeGreaterOrEqualTo: 1
  unit: rows
  description: Root-level canonicalUrl entry required.
```

**Data contract:**
```yaml
authoritativeDefinitions:
  - type: businessDefinition
    url: https://wiki.acme.com/customer-data
  - type: canonicalUrl
    url: https://github.com/acme/contracts/product-catalog.odcs.yaml
```

**Evaluation:**

1. Navigate to `authoritativeDefinitions` → 2 items
2. `arguments: { where: { field: type, in: [canonicalUrl] } }` → filter → 1 item
3. `metric: rowCount` → count = 1
4. `mustBeGreaterOrEqualTo: 1` → is 1 >= 1?
5. **Result: PASS** (actual: 1 row)

#### Example 6: arguments with `distinct`

**Criterion:**
```yaml
- id: L5-C06
  property: quality[].dimension
  type: library
  metric: rowCount
  arguments:
    distinct: true
    validValues:
      - accuracy
      - completeness
      - conformity
      - consistency
      - coverage
      - timeliness
      - uniqueness
  mustBeGreaterOrEqualTo: 3
  unit: rows
  description: ">= 3 of 7 Data QoS dimensions represented across all quality rules."
```

**Data contract:**
```yaml
schema:
  - name: product
    properties:
      - name: product_id
        quality:
          - metric: nullValues
            mustBe: 0
            dimension: completeness
          - metric: duplicateValues
            mustBe: 0
            dimension: uniqueness
      - name: unit_price
        quality:
          - metric: invalidValues
            mustBe: 0
            dimension: conformity
      - name: updated_at
        quality:
          - metric: nullValues
            mustBe: 0
            dimension: completeness
          - metric: invalidValues
            mustBe: 0
            dimension: timeliness
```

**Evaluation:**

1. Collect all `dimension` values across all quality rules → `[completeness, uniqueness, conformity, completeness, timeliness]`
2. `arguments: { validValues: [...] }` → filter to allowed values → all 5 match
3. `arguments: { distinct: true }` → deduplicate → `[completeness, uniqueness, conformity, timeliness]` → 4
4. `metric: rowCount` → count = 4
5. `mustBeGreaterOrEqualTo: 3` → is 4 >= 3?
6. **Result: PASS** (actual: 4 distinct dimensions)

#### Example 7: arguments with `relatesTo` (cross-artifact)

**Criterion:**
```yaml
- id: L3-P06
  property: outputPorts[].contractId
  type: library
  metric: rowCount
  arguments:
    relatesTo: "[DataContract].status"
    where:
      field: status
      in: [active]
  mustBeGreaterOrEqualTo: 1
  unit: rows
  description: >
    At least one outputPort.contractId references an ODCS
    DataContract with status: active.
```

**Data product:**
```yaml
outputPorts:
  - name: rawtransactions
    contractId: c2798941-1b7e-4b03-9e0d-955b1a872b32
  - name: consolidatedtransactions
    contractId: a44978be-1fe0-4226-b840-1b715bc25c63
```

**Linked ODCS contracts:**
```yaml
# c2798941-1b7e-4b03-9e0d-955b1a872b32
kind: DataContract
id: c2798941-1b7e-4b03-9e0d-955b1a872b32
status: draft
```
```yaml
# a44978be-1fe0-4226-b840-1b715bc25c63
kind: DataContract
id: a44978be-1fe0-4226-b840-1b715bc25c63
status: active
```

**Evaluation:**

1. Collect all `contractId` values → `[c2798941..., a44978be...]`
2. `arguments: { relatesTo: "[DataContract].status" }` → for each contractId, resolve the linked DataContract and read its `status`:
   - `c2798941...` → `status: draft`
   - `a44978be...` → `status: active`
3. `arguments: { where: { field: status, in: [active] } }` → filter → 1
4. `metric: rowCount` → count = 1
5. `mustBeGreaterOrEqualTo: 1` → is 1 >= 1?
6. **Result: PASS** (actual: 1 active contract)

#### Example 8: Policy evaluation

**Policy:**
```yaml
policies:
  - id: POL-P01
    name: statusCap
    property: status
    type: library
    metric: validValues
    arguments:
      caps:
        retired: 0
        deprecated: 2
        proposed: 1
        draft: 1
        active: 5
    description: >
      Status determines the maximum achievable maturity level.
```

**Data product:**
```yaml
status: deprecated
```

**Evaluation:**

1. Policy is evaluated before any level criteria
2. Read `property`: `status` → `deprecated`
3. `arguments: { caps: { deprecated: 2 } }` → maximum level = 2
4. The evaluator proceeds with level criteria but caps the result at L2
5. Even if all L5 criteria pass, the product scores **L2**

#### Example 9: arguments with `anyOf`

**Criterion:**
```yaml
- id: L4-C10
  property: schema[].properties[]
  type: library
  metric: rowCount
  arguments:
    anyOf: [primaryKey, unique]
  mustBeGreaterOrEqualTo: 1
  unit: rows
  description: At least one property declares primaryKey or unique.
```

**Data contract:**
```yaml
schema:
  - name: product
    properties:
      - name: product_id
        primaryKey: true
      - name: unit_price
      - name: sku
        unique: true
      - name: category
```

**Evaluation:**

1. Iterate all properties → 4 items
2. `arguments: { anyOf: [primaryKey, unique] }` → filter to items that have at least one → 2 items (`product_id`, `sku`)
3. `metric: rowCount` → count = 2
4. `mustBeGreaterOrEqualTo: 1` → is 2 >= 1?
5. **Result: PASS** (actual: 2 rows)

### 7. Structured Example

A more complete example showing `arguments` with `relatesTo`, `where`, `having`, `distinct`, and `anyOf`:

```yaml
kind: MaturityModel
apiVersion: v0.2.0
id: 3c8a1f47-6d9e-4a2b-b5f3-9e2d4c7a8b1f
name: odps-data-product-maturity-model
version: 1.0.0
status: active
description:
  purpose: Maturity model for ODPS data products.
  limitations: >
    Criteria are designed for ODPS v1.0.0. The "latest" apiVersion
    gate at higher levels tracks future ODPS releases.
  usage: >
    Evaluate a data product YAML file against each level's criteria
    sequentially. Policies are evaluated first.
tags:
  - maturity-model
  - odps
targetApplication: ODPS
targetApiVersion: v1.0.0

policies:
  - id: POL-P01
    name: statusCap
    property: status
    type: library
    metric: validValues
    arguments:
      caps:
        retired: 0
        deprecated: 2
        proposed: 1
        draft: 1
        active: 5
    description: >
      Status determines the maximum achievable maturity level.

  - id: POL-P02
    name: selfSourcing
    property: inputPorts
    type: library
    metric: validValues
    arguments:
      where:
        field: name
        in: [self]
      contractIdMustMatch: product.id
      versionMustMatch: product.version
    description: >
      Data products that generate data without external sources
      must use the "self" convention.

levels:
  - level: 1
    name: Structural
    description: Minimum valid ODPS file.
    criteria:
      - id: L1-P01
        property: apiVersion
        type: library
        metric: validValues
        mustBe: v1.0.0
        description: ODPS standard version is v1.0.0.

      - id: L1-P02
        property: kind
        type: library
        metric: validValues
        mustBe: DataProduct
        description: kind must equal DataProduct.

      - id: L1-P05
        property: outputPorts
        type: library
        metric: rowCount
        mustBeGreaterOrEqualTo: 1
        unit: rows
```

## Alternatives

### 1. No standard: keep maturity model syntax implicit

Rejected because two models (ODCS, ODPS) need maturity assessment and a third is likely. Without a shared spec, divergence is inevitable.

### 2. Policies as customProperties

Rejected because status overrides and conventions are core model behavior, not "custom." They deserve first-class syntax.

### 3. Scope as a dispatch mechanism

Early experimentation used a `scope` field on criteria to route evaluation to specific functions. This was replaced by `metric` + `arguments`, aligning with the ODCS data quality vocabulary and eliminating a concept with no equivalent in the existing standards.

### 4. relatesTo and condition as top-level criterion fields

Considered as dedicated fields on criteria. Rejected in favor of placing them inside `arguments` to keep the criterion structure aligned with ODCS data quality and avoid introducing fields that have no counterpart in the existing vocabulary.

### 5. Rename relatesTo to avoid collision with RFC-0031

Considered but rejected. RFC-0031's `relatesTo` operates within ODCS schema relationships (data lineage, business context). OMMS's `relatesTo` operates as an argument within maturity model criteria (evaluation dependencies). The contexts are distinct.

### 6. New metrics (percentWith, distinctCount)

Considered introducing new metric names. Rejected in favor of extending existing metrics (`rowCount`, `nullValues`) with arguments (`having`, `anyOf`, `where`, `distinct`). This avoids growing the metric vocabulary unnecessarily.

## Decision

> The decision made by the TSC.

## Consequences

**If adopted:**

- OMMS becomes a formal Bitol standard alongside ODCS and ODPS.
- Bumps to OMMS v1.0.0.
- Maturity models for ODCS and ODPS are authored using OMMS v1.0.0.
- Evaluation tooling gains a formal contract for how to interpret maturity model YAML files.
- Future maturity models (e.g., for data mesh architectures, data governance programs) can use the same standard.

**Migration from experimental models:**

Those are internal notes for prior experimentations:

- `rule` → `metric`
- `scope` → removed, replaced by `metric` + `arguments`
- `unit: items`, `unit: errors`, `unit: members`, `unit: dimensions` → `unit: rows`
- `policies` and `arguments` are additive
- Models should set `apiVersion` to `v0.2.0`
