## Summary

This RFC introduces OOCS (Open Orchestration and Control Standard), a declarative YAML format (`kind: OrchestrationControl`) for expressing orchestration directives and control policies for data products and their sidecars. An OOCS document answers: *given this data product, what actions should be taken, when, by whom, and under what conditions?*

OOCS fills the operational gap between the existing Bitol standards:

| Standard | Answers                       | Analogy               |
| -------- | ----------------------------- | --------------------- |
| ODCS     | What does the data look like? | Blueprint             |
| ODPS     | What is the data product?     | Bill of materials     |
| OORS     | What happened?                | Inspection report     |
| **OOCS** | **What should happen?**       | **Operations manual** |

## Motivation

ODCS defines what data looks like (contracts). ODPS defines what a data product is (ports, teams, identity). OORS captures what happened (observability results). None of these define **what should happen next** — the orchestration and control instructions that govern a data product's operational behavior.

Without a standard for operational directives:

- Each data product sidecar invents its own configuration format, preventing interoperability.
- Control centers cannot reason about orchestration across data products because there is no shared vocabulary.
- Operational intent is buried in imperative code (Airflow DAGs, shell scripts) rather than declared alongside the data product specification.

OOCS fills this gap with a declarative, transport-agnostic format that is contract-aware and composable.

This aligns with the guiding value of *favoring interoperability over readability* — OOCS is designed for machine consumption by sidecars and control planes, with human readability as a secondary benefit.

## Design and examples

### Mental model: request and response

An OOCS document is a **set of requests**. Each action declares an intent: "register this data product", "measure SLA compliance", "run data quality checks." The emitter sends the OOCS document (or individual actions within it) to a target system.

The `response` field on each action tells the target system **how to notify the emitter about the outcome**. This is the feedback channel — a Kafka topic to publish results to, or a REST endpoint to call back with the result.

```
Emitter                                    Target System
   │                                             │
   │── OOCS action (request) ───────────────────►│
   │                                             │  (executes the action)
   │◄── result via response channel using OORS ──│
   │                                             │
```

For example, a sidecar sends a `register` action to the control center. The `response` field says "send the acknowledgment to Kafka topic `controlcenter.dataproduct.feedback`." Or a control center sends a `measure` action; the `response` field says "post the SLA results to this Actian Data Observability REST endpoint."

### Design principles

1. **Declarative** — Describe desired state, not imperative steps. The sidecar or control plane decides how to achieve it.
2. **Contract-aware** — First-class (but optional) linkage to ODCS and ODPS identifiers.
3. **Response-oriented** — Each action declares how the emitter expects to be notified about the outcome.
4. **Lightweight** — Flat structure, minimal required fields, easy to author by hand.
5. **Kubernetes-inspired** — Uses `kind`, `apiVersion`, `id`, `status` resource model consistent with ODCS and ODPS.
6. **Composable** — Multiple directives in a single document; each directive is independently actionable.

### Specification

**Version**: v0.1.0 (draft)
**Media type**: `application/oocs+yaml;version=0.1.0`
**File extension**: `.oocs.yaml`

### Fundamentals (Top-level structure / header)

An OOCS document is a YAML file with the following top-level fields:

| Field                                    | Type            | Required | Description                                                               |
| ---------------------------------------- | --------------- | -------- | ------------------------------------------------------------------------- |
| `kind`                                   | string          | yes      | Always `OrchestrationControl`                                             |
| `apiVersion`                             | string          | yes      | OOCS standard version, e.g. `v0.1.0`                                      |
| `id`                                     | string (UUID)   | yes      | Unique identifier for this document                                       |
| `version`                                | string (semver) | yes      | Document version (e.g. `v1.0.0`). Must be bumped when directives change   |
| `status`                                 | enum            | no       | Document maturity: `proposed`, `draft`, `active`, `suspended`, `retired`  |
| `name`                                   | string          | no       | Human-readable label for the document                                     |
| `domain`                                 | string          | no       | Business domain (e.g. `seller`, `finance`)                                |
| `tenant`                                 | string          | no       | Organization or tenant name                                               |
| `description`                            | object          | no       | Free-text description block                                               |
| `description.purpose`                    | string          | no       | What this document is for                                                 |
| `description.limitations`                | string          | no       | What this document does not cover                                         |
| `actions`                                | array           | no       | Named action definitions (see Actions below)                              |
| `tags`                                   | array of string | no       | Free-form labels for filtering and grouping                               |
| `customProperties`                       | array           | no       | Arbitrary key-value pairs for extensibility                               |
| `customProperties[].property`            | string          | yes*     | Property name (*required within each entry)                               |
| `customProperties[].value`               | string          | yes*     | Property value (*required within each entry)                              |
| `authoritativeDefinitions`               | array           | no       | Links to external authoritative sources                                   |
| `authoritativeDefinitions[].type`        | string          | yes*     | Definition type (e.g. `businessDefinition`, `runbook`) (*required within) |
| `authoritativeDefinitions[].url`         | string (URL)    | yes*     | URL to the external resource (*required within each entry)                |
| `authoritativeDefinitions[].description` | string          | no       | What the linked resource contains                                         |

### Actions

Named action definitions that the sidecar or control plane can execute. Each action has a `type` that determines its behavior, a `payload` that carries the request data, and a `response` that specifies how the emitter expects to be notified about the outcome.

#### Common action fields

| Field                       | Type   | Required    | Description                                                        |
| --------------------------- | ------ | ----------- | ------------------------------------------------------------------ |
| `actions[].id`              | string | yes         | Unique identifier within this document. Used for cross-referencing |
| `actions[].name`            | string | yes         | Human-readable name                                                |
| `actions[].type`            | enum   | yes         | Action type: `register`, `heartbeat`, `deregister`, `configure`, `measure`, `dataQuality` |
| `actions[].description`     | string | no          | What this action does                                              |
| `actions[].response`       | object | yes         | How the emitter is notified about the outcome                      |
| `actions[].response.type`  | enum   | yes         | Response channel: `kafka`, `rest`                                  |
| `actions[].response.topic` | string | conditional | Kafka topic for results. Required when `response.type` is `kafka`  |
| `actions[].response.url`   | string | conditional | REST callback URL for results. Required when `response.type` is `rest` |

#### Action type: `register`

Sent on startup to announce a data product to the control center.

| Field                       | Type            | Required | Description                                  |
| --------------------------- | --------------- | -------- | -------------------------------------------- |
| `actions[].payload`         | object          | yes      | Identifies the data product being registered |
| `actions[].payload.kind`    | string          | yes      | Artifact type (e.g. `DataProduct`)           |
| `actions[].payload.id`      | string (UUID)   | yes      | Unique identifier of the data product        |
| `actions[].payload.version` | string (semver) | yes      | Version of the data product                  |

#### Action type: `heartbeat`

Periodic liveness signal to the control center.

| Field                      | Type    | Required | Description                                          |
| -------------------------- | ------- | -------- | ---------------------------------------------------- |
| `actions[].payload`       | object  | yes      | How often the heartbeat is sent                      |
| `actions[].payload.value` | integer | yes      | Numeric interval value                               |
| `actions[].payload.unit`  | enum    | yes      | Time unit: `s` (seconds), `m` (minutes), `h` (hours) |

#### Action type: `deregister`

Sent on graceful shutdown to remove a data product from the control center.

| Field                       | Type            | Required | Description                                    |
| --------------------------- | --------------- | -------- | ---------------------------------------------- |
| `actions[].payload`         | object          | yes      | Identifies the data product being deregistered |
| `actions[].payload.kind`    | string          | yes      | Artifact type (e.g. `DataProduct`)             |
| `actions[].payload.id`      | string (UUID)   | yes      | Unique identifier of the data product          |
| `actions[].payload.version` | string (semver) | yes      | Version of the data product                    |

#### Action type: `configure`

Sent by the control center to a data product to deliver configuration content (e.g., the ODPS specification).

| Field                       | Type            | Required | Description                                     |
| --------------------------- | --------------- | -------- | ----------------------------------------------- |
| `actions[].payload`         | object          | yes      | Configuration payload for the data product      |
| `actions[].payload.kind`    | string          | yes      | Artifact type (e.g. `DataProduct`)              |
| `actions[].payload.id`      | string (UUID)   | yes      | Unique identifier of the data product           |
| `actions[].payload.version` | string (semver) | yes      | Version of the data product                     |
| `actions[].payload.content` | string          | yes      | The configuration content (e.g. full ODPS YAML) |

#### Action type: `measure`

Requests a measurement (e.g. SLA compliance, freshness, completeness) for a governed artifact. The results are posted back via the `response` channel, typically to an observability platform.

| Field                       | Type            | Required | Description                                         |
| --------------------------- | --------------- | -------- | --------------------------------------------------- |
| `actions[].payload`         | object          | yes      | Identifies the artifact to measure                  |
| `actions[].payload.kind`    | string          | yes      | Artifact type (e.g. `DataContract`)                 |
| `actions[].payload.id`      | string (UUID)   | yes      | Unique identifier of the artifact                   |
| `actions[].payload.version` | string (semver) | yes      | Version of the artifact                             |

#### Action type: `dataQuality`

Requests data quality checks for a governed artifact. The results are posted back via the `response` channel, typically to a data quality or remediation tool.

| Field                       | Type            | Required | Description                                         |
| --------------------------- | --------------- | -------- | --------------------------------------------------- |
| `actions[].payload`         | object          | yes      | Identifies the artifact to validate                 |
| `actions[].payload.kind`    | string          | yes      | Artifact type (e.g. `DataContract`)                 |
| `actions[].payload.id`      | string (UUID)   | yes      | Unique identifier of the artifact                   |
| `actions[].payload.version` | string (semver) | yes      | Version of the artifact                             |

### Example 1: Minimal — startup registration and heartbeat

```yaml
kind: OrchestrationControl
apiVersion: v0.1.0
id: 8a3e4b10-2f7c-4d1a-9e5b-6c8f0a1d3e7b
version: v1.0.0

actions:
  - id: register
    name: Register with control center
    type: register
    description: Announce this data product to the control center on startup.
    response:
      type: kafka
      topic: controlcenter.dataproduct.feedback
    payload:
      kind: DataProduct
      id: fbe8d147-28db-4f1d-bedf-a3fe9f458427
      version: v0.1.0

  - id: heartbeat
    name: Periodic liveness check
    type: heartbeat
    description: Send a heartbeat every 30 seconds.
    response:
      type: kafka
      topic: controlcenter.dataproduct.feedback
    interval:
      value: 30
      unit: s
```

### Example 2: Structured — shutdown with metadata

```yaml
kind: OrchestrationControl
apiVersion: v0.1.0
id: 9b4f5c21-3a8d-4e2b-af6c-7d9e1b2c4f8a
version: v1.0.0
status: active
name: customer-dp-shutdown
domain: seller
tenant: RetailCorp

description:
  purpose: Graceful shutdown directives for the customer data product.
  limitations: Does not handle crash recovery — see runbook.

actions:
  - id: deregister
    name: Deregister from control center
    type: deregister
    description: Notify the control center on graceful shutdown.
    response:
      type: kafka
      topic: controlcenter.dataproduct.feedback
    payload:
      kind: DataProduct
      id: fbe8d147-28db-4f1d-bedf-a3fe9f458427
      version: v0.1.0

tags:
  - shutdown
  - graceful

authoritativeDefinitions:
  - type: runbook
    url: https://wiki.example.com/customer-dp/shutdown-runbook
    description: Detailed shutdown and recovery procedures.
```

### Example 3: Control center to data product — configure

The control center pushes the ODPS specification to the data product sidecar. The OOCS envelope wraps the full ODPS YAML content.

```yaml
kind: OrchestrationControl
apiVersion: v0.1.0
id: a1b2c3d4-5e6f-7a8b-9c0d-1e2f3a4b5c6d
version: v1.0.0

actions:
  - id: configure
    name: Deliver ODPS specification
    type: configure
    response:
      type: kafka
      topic: controlcenter.dataproduct.command
    payload:
      kind: DataProduct
      id: fbe8d147-28db-4f1d-bedf-a3fe9f458427
      version: v2.0.0
      content: |
        kind: DataProduct
        apiVersion: v1.0.0
        id: fbe8d147-28db-4f1d-bedf-a3fe9f458427
        version: v2.0.0
        name: Customer Data Product
        domain: seller
        status: active
        outputPorts:
        - name: rawtransactions
          version: v2.0.0
          contractId: c2798941-1b7e-4b03-9e0d-955b1a872b33
```

### Example 4: SLA measurement via Actian Data Observability

A control center requests SLA compliance measurement for a data contract. The results are posted back to Actian Data Observability's REST endpoint.

```yaml
kind: OrchestrationControl
apiVersion: v0.1.0
id: d4e5f6a7-8b9c-0d1e-2f3a-4b5c6d7e8f9a
version: v1.0.0
name: customer-contract-sla-check
domain: seller
tenant: RetailCorp

actions:
  - id: measure-sla
    name: SLA compliance measurement
    type: measure
    description: Measure SLA compliance for the customer data contract and report results to Actian Data Observability.
    response:
      type: rest
      url: https://actian-dataobservability.retailcorp.com/api/v1/sla/results
    payload:
      kind: DataContract
      id: c2798941-1b7e-4b03-9e0d-955b1a872b33
      version: v2.0.0
```

### Example 5: Data quality request to a custom tool

A sidecar requests data quality checks for a contract. The results are sent to an internal tool (JoeThePlumberFixer) via REST callback.

```yaml
kind: OrchestrationControl
apiVersion: v0.1.0
id: f1a2b3c4-5d6e-7f8a-9b0c-1d2e3f4a5b6c
version: v1.0.0
name: customer-dq-check
domain: seller

actions:
  - id: dq-check
    name: Data quality validation
    type: dataQuality
    description: Run data quality rules defined in the contract and send results to JoeThePlumberFixer for remediation.
    response:
      type: rest
      url: https://joetheplumberfixer.internal.retailcorp.com/api/v1/dq/ingest
    payload:
      kind: DataContract
      id: c2798941-1b7e-4b03-9e0d-955b1a872b33
      version: v2.0.0
```

## Open issues and discussion

### OI-1: Additional action types

The current specification defines four action types (`register`, `heartbeat`, `deregister`, `configure`). The reference implementation also uses:

- `deliver-odcs` — Delivers individual ODCS contracts to the sidecar.
- `deliver-spec` — Wraps ODPS/ODCS specs from GitOps pipelines.
- `announce` — Variant of `register` used in some flows.
- `request-odcs` — Sidecar requests specific contracts from the control center.
- `ready` — Sidecar signals it has received all contracts and is operational.

**Question**: Should all of these be formalized as standard action types, or should some remain tool-specific? Should the `type` field be an open enum (allowing custom action types via `customProperties`) or a closed enum?

**Recommendation**: Define a core set of standard action types (`register`, `heartbeat`, `deregister`, `configure`, `deliver`, `request`, `ready`) and allow custom types prefixed with `x-` (e.g. `x-audit-trigger`).

### OI-2: Bidirectionality

OOCS documents currently describe actions from a single perspective (either the sidecar or the control center). Should the standard include a `direction` field (`upstream` for sidecar-to-CC, `downstream` for CC-to-sidecar) or remain direction-agnostic?

**Recommendation**: Direction-agnostic. The response topic or endpoint implicitly encodes direction. Adding a `direction` field would duplicate information and risk inconsistency.

### OI-3: Conditions and guards

The current design is purely declarative with no conditional logic. Should OOCS support conditions on actions (e.g. "only send heartbeat if status is active", "deregister if health check fails 3 times")?

**Recommendation**: Not in v0.1.0. Conditional logic belongs in the sidecar implementation, not the declarative specification. If demand warrants it, a future version could add an optional `when` clause using a simple expression language.

### OI-4: Security and authentication

The specification does not address authentication for REST responses or authorization for Kafka topics. Should OOCS include security-related fields (e.g. `response.auth.type`, `response.auth.credentials`)?

**Recommendation**: Not in the spec itself. Security configuration is deployment-specific (SASL_SSL for Kafka, OAuth2 for REST) and should be handled at the infrastructure level, not embedded in OOCS documents which may be stored in Git.

### OI-5: JSON Schema

Should a companion JSON schema (`oocs-json-schema-v0.1.0.json`) be published alongside this RFC, following the ODCS and ODPS precedent?

**Recommendation**: Yes. A JSON schema enables IDE validation, linting, and tooling integration. It should be published in the `open-data-contract-standard` repository or a new `open-orchestration-control-standard` repository.

### OI-6: Relationship to OORS

OOCS drives actions; OORS captures results. Should OOCS include an explicit linkage to OORS (e.g. an action type `observe` that references an OORS template), or should the relationship remain implicit?

**Recommendation**: Implicit for v0.1.0. The sidecar produces OORS documents as a side effect of executing OOCS actions, but the OOCS document itself does not need to reference OORS templates.

### OI-7: OMDS applicability

RFC-0039 (OMDS) defines `sourceKind` for `DataContract` and `DataProduct`. Should OMDS also support `sourceKind: OrchestrationControl` for diffing OOCS documents? If so, what impact classification rules apply?

**Recommendation**: Yes, as a future extension to OMDS. Suggested impact rules: removing an action = MAJOR, changing response = MAJOR, changing interval = MINOR, changing description = PATCH.

## Alternatives

### 1. Embed orchestration in ODPS

ODPS could be extended with an `orchestration` section containing action definitions. Rejected because it conflates identity/structure (what a data product *is*) with operational behavior (what it *does*), violating separation of concerns. A data product's schema and ports rarely change, but its orchestration directives may be updated frequently.

### 2. Use existing workflow engines (Airflow DAGs, Argo Workflows)

These are imperative and tightly coupled to specific runtimes. OOCS is declarative and transport-agnostic, letting the sidecar or control plane choose the execution strategy. A workflow engine may *implement* OOCS actions, but the directives themselves should be runtime-independent.

### 3. Ad-hoc configuration per sidecar

Each sidecar could define its own configuration format. Rejected because it prevents interoperability and makes the control center unable to reason about orchestration across data products from different teams or organizations.

## Decision

> The decision made by the TSC.

## Consequences

- Introduces a new standard in the Bitol ecosystem (`kind: OrchestrationControl`) alongside ODCS, ODPS, and OORS.
- Sidecars and control centers gain a shared vocabulary for orchestration, enabling cross-product automation.
- Requires a companion JSON schema and parser libraries (reference implementation exists in `ai.jgp.dataproduct.sidecar`).
- Future OOCS versions may add action types beyond the initial set.
- The reference implementation in `ai.jgp.controlcenter.svc` and `ai.jgp.dataproduct.sidecar` already implements OOCS v0.1.0 and can serve as the basis for conformance testing.
- RFC-0039 (OMDS) should be extended in a future revision to support `sourceKind: OrchestrationControl`.
