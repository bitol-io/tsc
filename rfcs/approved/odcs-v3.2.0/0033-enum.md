# RFC 0033: Enum

**Champion:** Jochen Christ

**Authors:** Patrick Beitsma, Jochen Christ and Jean-Georges Perrin.


## Summary

Add native support for enumeration (enum) constraints in ODCS to specify that a property's value must be one of a predefined set of allowed values.

## Motivation

Enumerations are a fundamental data constraint used across all data platforms and programming languages.

Currently, ODCS lacks native enum support.

## Design and Examples

Add an `enum` property, similiar to JSON-Schema:

### Examples

**String enum:**
```yaml
- name: status
  logicalType: string
  enum:
    - pending
    - processing
    - shipped
    - cancelled
```

**Integer enum:**
```yaml
- name: priority
  logicalType: integer
  enum:
    - 1
    - 2
    - 3
```

## Alternatives

### Option A: Property-level `enum`
```yaml
- name: status
  logicalType: string
  required: false      # null values allowed
  enum:
    - pending
    - processing
```
Consistent with JSON Schema. Treats enum as a value constraint like `required` or `unique`.

### Option B: Inside `logicalTypeOptions` (rejected)
```yaml
- name: status
  logicalType: string
  logicalTypeOptions:
    enum:
      - pending
      - processing
```
Groups all type-related constraints together. Keeps the property level cleaner.

Rejected because we do not always have logical type.

### Option C: Enums as Objects

```yaml
- name: priority
  logicalType: integer
  required: false      # null values allowed
  enum:
    - value: 1
      label: One
      id: 1
      description: Highest ranked
      customProperties:
        - property: languageDe
          value: Eins
      authoritativeDefintions: //...
    - value: 2
      label: Two
      id: 2
      description: Highest ranked
    - value: 3
      label: Three
      id: 3
      description: Lowest ranked
```

Only `value` is required. Other fields are optional.
More powerful. But also more "complicated".

### Option D: Combine A and C

Allow both structures.


### Quality check with `validValues`
```yaml
- name: status
  logicalType: string
  quality:
    - type: library
      metric: invalidValues
      validValues:
        - pending
        - processing
      mustBeLessThan: 5
      unit: percent
```
Uses existing quality framework. Already available with ODCS 3.1.0

## Decision

We go for option C - march 2026.

## Consequences

- Native support for a fundamental data constraint
- Schema validation ensures non-empty, unique values
- Enables tooling to generate validation, documentation, and UI dropdowns
- Schema version bump required (minor version)
