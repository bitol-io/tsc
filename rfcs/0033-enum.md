# RFC 0033: Enum

**Champion:** Jochen Christ

## Summary

Add native support for enumeration (enum) constraints in ODCS to specify that a property's value must be one of a predefined set of allowed values.

## Motivation

Enumerations are a fundamental data constraint used across all data platforms and programming languages.

Currently, ODCS lacks native enum support.

## Design and Examples

Add an `enum` property, similiar to JSON-Schema:

### Examples

**String enum:**
```yaml
- name: status
  logicalType: string
  enum:
    - pending
    - processing
    - shipped
    - cancelled
```

**Integer enum:**
```yaml
- name: priority
  logicalType: integer
  enum:
    - 1
    - 2
    - 3
```

## Alternatives

### Option A: Property-level `enum`
```yaml
- name: status
  logicalType: string
  enum:
    - pending
    - processing
```
Consistent with JSON Schema. Treats enum as a value constraint like `required` or `unique`.

### Option B: Inside `logicalTypeOptions`
```yaml
- name: status
  logicalType: string
  logicalTypeOptions:
    enum:
      - pending
      - processing
```
Groups all type-related constraints together. Keeps the property level cleaner.


### Option C: Quality check with `validValues`
```yaml
- name: status
  logicalType: string
  quality:
    - type: library
      metric: invalidValues
      validValues:
        - pending
        - processing
      mustBe: 0
```
Uses existing quality framework. No schema changes needed, but more verbose.

## Decision

*The decision made by the TSC.*

## Consequences

- Native support for a fundamental data constraint
- Schema validation ensures non-empty, unique values
- Enables tooling to generate validation, documentation, and UI dropdowns
- Schema version bump required (minor version)
